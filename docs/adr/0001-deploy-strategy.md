# ADR-0001: Выбор стратегии безопасного развёртывания ML-сервиса

**Статус:** Accepted, 10.05.2026
**Автор:** Кузнецова П. Г.

## Контекст

ML-сервис на FastAPI отдаёт `/predict` и `/health`, регулярно обновляется новой версией модели или кода. В текущей реализации полностью отсутствует обработка ошибок: исключение в `predict` обрушит обработчик, а кривой формат входа уйдёт в 500. Без аккуратного выката любая регрессия моментально становится инцидентом на проде. Нужна стратегия, которая ограничивает blast radius выкатки, даёт быстрый rollback и не требует переписывать сервис под себя

## Рассмотренные варианты

| Стратегия     | Плюсы                                              | Минусы                                              | Риски                                                          |
|---------------|----------------------------------------------------|-----------------------------------------------------|----------------------------------------------------------------|
| Single deploy | проще некуда                                       | downtime, нет атомарного отката                     | при ошибке сервис лежит до ручного отката                       |
| Rolling       | без downtime, дешёво по ресурсам                   | смешанный трафик во время катки, медленный rollback | две версии отвечают одновременно — несогласованные предсказания  |
| **Blue-Green**| мгновенное переключение, мгновенный rollback       | x2 ресурсов на время катки                          | после переключения 100% трафика сразу на новой версии            |
| Canary        | постепенный прогрев, минимальный риск              | нужен балансировщик и онлайн-метрики качества       | без метрик качества — слепой выкат                               |
| Shadow        | нулевое влияние на пользователя, реальный трафик   | вдвое больше железа, сложная синхронизация запросов | пользователь не получает ответ новой версии — это не релиз       |

## Решение

Базовая стратегия — **Blue-Green** через два compose-файла (`docker-compose.blue.yml` v1.0.0 и `docker-compose.green.yml` v1.1.0) и nginx-балансировщик с `proxy_pass` на активный апстрим. Тот же nginx умеет в Canary через `split_clients 90/10`, поэтому миграция на Canary тривиальна, как только появятся онлайн-метрики качества предсказаний

## Последствия

- плюс: каждый релиз ставит образ на «полку» рядом с работающей версией, smoke-тест по `/health` и `/predict` идёт до переключения
- плюс: rollback — это `nginx -s reload` с предыдущим upstream, секунды
- минус: на момент катки в кластере живут две полные копии сервиса
- риск: после переключения вся нагрузка идёт на новую версию — гасим smoke-тестом и алертом на 5xx на первых 60 секундах
- риск: рассинхрон состояния — для текущего stateless классификатора неактуален, но при добавлении кеша/очередей решение нужно пересмотреть
